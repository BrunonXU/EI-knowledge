A Simple Pick-and-Place

机器人把方块放进托盘，看起来只是一个动作

但在系统里面，其实要先后回答两个不同的问题。

第一件事 · 它想怎么做？

模型先看懂任务

画面 + 指令 → VLA / WAM → 手该往哪里走

它要认出方块和托盘，理解“抓起、移动、放下”，最后给出 EEF 或 Joint 动作。

我们在意：模型到底有没有学会这个任务？

→

第二件事 · 这台机器人怎么做到？

机器人再用自己的身体执行

模型动作 → 执行接口 → 真实关节运动

不同机器人手臂长度、关节数量和活动范围都不同，但都要把方块安全放进同一个托盘。

我们在意：它能不能按自己的身体条件把任务做成？

这套 PPT 只围绕这条主线：上层学任务，下层负责把任务做出来。

Terminology

先分清：状态表示、动作表示与执行接口

EEF 描述「手去哪」，Joints 描述「各关节怎么动」—— 两者不是维度之争，而是抽象层级之争

EEF / Task-space action

[Δx, Δy, Δz, Δroll, Δpitch, Δyaw, g] ∈ ℝ7

这里讨论的是末端 6-DoF 位姿增量 + 夹爪值；EEF frame 本身是坐标系，二者不要混用

Joints（关节空间）

q ∈ ℝn （G2 上半身 n = 14）

以智元 G2 为例：双臂各 7 关节，数据即关节角向量

FK（正运动学）

FK: q → EEF 位姿

确定性、唯一解

IK（逆运动学）

IK: EEF 位姿 → q

可能多解、需选择策略 —— 后文讨论的关键环节

IK / 控制器的角色：把任务空间目标变成可执行命令

Topic One

Three Viewpoints

核心矛盾：任务语义通用性 vs 本体执行精度

01
Joint 保留本体运动先验 示范里的肘部偏好、连续 IK 分支与限位习惯都被保留下来；固定本体控制稳定，但跨本体迁移受限
02
EEF 抽象任务几何 更接近“手应该去哪”，但没有完整表达“胳膊如何过去”；可达性、奇异点与多解仍由下游处理
03
EEF + 关节状态融合 EEF 提供共享任务几何，关节状态暴露当前本体构型；代表性方向：GEAR-VLA 的 embodiment canonicalization

这不是二选一：真正要选的是“策略抽象层级 + 执行接口”的组合 →

One Task, Two Responsibilities

先别急着比较模型：先看它负责哪一段

同一个抓放任务，可以清楚地拆成“想清楚怎么做”和“真的让机器人做到”。

理解任务，给出动作意图

看到画面和指令
→ 找到方块与托盘
→ 预测手该怎么移动

这里关心的不是电机怎么转，而是模型有没有理解抓取、搬运和放置，以及动作会让世界发生什么变化。

输出可以是：EEF，也可以是 Joint

→ 明确的
动作接口

下层 · Robot Execution

结合自己的身体，把动作做出来

接收模型动作
→ 检查够不够得到、会不会撞
→ 变成真实关节命令

机器人 A 和机器人 B 可以采用不同关节姿态；只要都安全、稳定地把方块放进托盘，任务就是同一个。

最终输出：机器人真正能执行的关节运动

解耦的目的：让模型少记机器人身体细节，让执行层专门处理身体差异。

What Should the Model Predict?

模型输出什么：直接给关节，还是先说“手怎么走”

两条路线都能完成抓放；区别在于，模型需要记住多少机器人身体细节。

路线 A · 模型直接输出 Joint

直接告诉每个关节转多少

模型→关节角→机器人

好处：示范里的肘部姿态、关节习惯都保留下来，固定机器人上往往更直接。

代价：换一台关节数量或结构不同的机器人，原来的输出很难直接复用。

路线 B · 模型输出 EEF

先告诉“手应该往哪里走”

模型→手的目标→执行层

好处：“抓住方块并放进托盘”的手部运动，更容易让不同机器人共享。

代价：模型只说了手去哪，没有说清每个关节怎样配合，执行层必须补上。

如果目标是跨机器人共享，EEF 更像一份共同的“任务说明”；Joint 更像某一台机器人的“具体做法”。

Same Intent, Different Bodies

同一个“把方块放进托盘”，不同机器人可以有不同做法

上层共享的是手的目标；下层根据各自的身体，算出不同的关节动作。

模型给出的共同意图

“手向前移动，抓住方块，再移动到托盘上方并松开”

机器人 A · 单臂机械臂

自己的执行方法

手的目标→ 检查可达→ 6/7 个关节

它可能侧着肘部过去，只要不超限、不碰撞，并且夹爪到达目标即可。

同一
任务

机器人 B · 双臂 / 人形机器人

另一种执行方法

手的目标→ 协调身体→ 更多关节

它可能要同时调整肩、肘、腰或另一只手，但最终完成的是同一个物块放置任务。

理想的分工：模型负责“做什么”，机器人执行层负责“我具体怎么做”。

Paper Reference · Deep Dive

固定本体：Joint + chunk-wise delta 最优

跨本体迁移：EEF + chunk-wise delta 更适合

Joint 更强的三大原因

①

避免 IK 误差

IK 多解歧义即是误差来源，直接输出 Joint 绕开这一环节

②

数据本就在关节流形上

示范轨迹包含肘部偏好、连续 IK 分支、避限位习惯

③

强生成模型可建模多模态分布

Flow Matching / Diffusion 能表达多解，弱回归会平均化

局限性：EEF 依赖 IK 质量 / 跨本体实验规模小，未覆盖 force·tactile·灵巧手 / 未来可能按任务阶段动态混合而非二选一

Deep Dive · Delta

Chunk-wise delta 让整段目标锚定同一观测帧，避免表示层的逐步漂移

01Step-wise逐步增量
at = qt − qt−1 → qt = qt−1 + at

每个标签参考上一个目标，预测偏差可能在目标重建时逐步传递 —— 像每一步都用上一落点重新量距离

02Chunk-wise锚定增量
at+k = qt+k − qt → qt+k = qt + at+k

整段锚定同一观测点 qt，标签彼此独立、天然并行 —— 每个目标都从同一地标量起

03为什么 Delta 优于 Absolute

Absolute：直接输出 qtarget，不同机器人绝对值分布差异巨大 → 需拟合全局关节空间，泛化差

Delta：输出 Δq = qtarget − qcurrent，分布紧凑（均值≈0、方差小）→ 与起始状态无关，泛化强

04EEF 的特殊性SE(3) 上的 delta 不是加法
Ttarget = ΔT · Tcurrent （位置可加，旋转是矩阵 / 四元数乘法）

⚠ EEF delta 需要群运算，工程实现比 joint delta 复杂 —— 这也是固定本体 Joint delta 更优的原因之一

边界：这里讨论的是 action chunk 的标签参数化；真实执行通常仍会闭环重规划，因此不等同于“整段开环执行必然不漂移”。

Same Semantics, Different Names

三种命名，其实是同一种 anchor 逻辑

共同语义：一个 action chunk 内，所有未来 EEF target 都相对于当前观测时刻的 EE pose 表达，而不是相对前一个预测点。

01
UMI Relative Trajectory as Action Representation 论文原名强调 trajectory：整段未来 pose 相对于同一个 initial/current EE pose。
02

Two Different Problems

跨本体学习有两个不同的问题

GAP A · REPRESENTATION / TRANSFER

人类经验 → 可共享规律 → 机器人动作意图

差异：morphology、速度、grasp style、camera ego-motion

human experience ≠ robot execution

对应方法：relative EEF、canonical action、world supervision、EgoWAM

GAP B · EXECUTION

EEF / task intent → 本体约束 → feasible joints

约束：IK 多解、joint limit、collision、singularity、continuity、contact / balance

EEF intent → IK / OSC / WBC → q / q̇ / τ

对应方法：IK、OSC / WBC、IKSel、embodiment adapter

EgoWAM 处理“什么知识可以共享”；IKSel 处理“这台机器人怎样更快找到一个 IK 解”。

Controller Selection

我们应该如何选择合适的 IK / 控制器？

先确定控制层级与训练接口，再比较求解器；排行榜速度不是第一判断标准

01 · 策略到底输出什么？

Joint target → joint controller；EEF pose → IK / OSC；全身接触与平衡 → WBC / MPC。先选层级，不能拿一个 IK 包打天下。

02 · 数据由谁生成？

优先复用采集时的控制器、厂商验证接口或 benchmark 标准控制器，避免训练与部署的关节分布发生变化。

03 · 闭环真正关心什么？

连续性、joint limit / collision / singularity、求解失败恢复，以及目标控制频率下的稳定延迟。

IK Summary

三个核心认知

EEF 方案的上限，最终由运动学可达性与控制接口共同决定

一、EEF 到执行的必经桥梁

模型输出 EEF 后，仍需 IK / OSC / WBC 映射到关节或力矩；控制接口决定执行效果。

二、数据侧不可或缺

脚本采集、MimicGen 扩充、仿真生成（Isaac Sim）都依赖 IK 从 EEF 反算平滑关节轨迹。

三、选型与速度

多解歧义需合理种子策略（如 IKSel）；高频回路要求快且不牺牲解的质量。

IK 既决定执行质量，也是数据流水线的关键一环 —— 选对 IK、又快又稳，是 EEF 方案可用的最后一块拼图。

论文 3：IKSel

Inverse Kinematics Seed Selection for Fast IK Convergence · arxiv 2503.22234

核心：先建机器人的关节姿态 → 末端位姿运动学地图，再为 Numerical IK 选起点

① 采样关节空间在 joint limits 内均匀生成 qi
② 计算并标注xi=FK(qi)
同时保存 J(qi)+
③ 建 KDTree按 TCP pose xi 索引每个 qi
④ 在线查询输入 target x*
输出 seed q0
1

Workspace 先筛“近”：从地图中找一组 xi 接近 target x* 的候选；它们的手都在目标附近，但关节构型可能完全不同。

2

Jacobian 再预测“小”：对每个候选估计 Δqi=J(qi)+(x*−xi)；优先选择预计一步关节修正量 ‖Δqi‖ 最小的 seed。

3

Numerical IK 求解；失败后换“远”分支：若迭代撞到 joint limit / 局部极值，就在优质候选中改选关节构型差异更大的 seed，避开原来的 basin。

记住三个词：Workspace“近” → 预测 Δq“小” → 失败后 Joint Space“远”

IKSel 只选择更有希望收敛的初始关节构型 q0；Numerical IK 才计算最终解 q*。论文这里没有用 current q 做连续性排序。

它解决的问题

避免“末端最近但局部运动学很差”的 seed，让数值 IK 更快、更容易收敛。

01

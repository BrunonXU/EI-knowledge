主题一

Joint vs EEF 的本质探讨

Three Viewpoints
1
Joint 照葫芦画瓢
  • Joint 离开了本体换一个机器人，本质照葫芦画瓢
  • 不是最优的终局之解，只是现在工程上表现力比较好
  • 数据绑定特定机器人构型，泛化能力受限
2
EEF 能学任务本意但忽略 IK 局限
  • 学习 EEF 能学到任务本意
  • 但忽略本体 IK 局限性
  • 胳膊如何过去很重要，不能只在乎末端执行器如何到达和执行
  • IK 多解歧义导致执行时抖动或失败
3
EE + 关节状态的融合方案
  • 使用 EE 提供共享任务几何
  • 同时保留关节状态或本体配置来保证可执行性
  • GEAR-VLA 即为此方向的代表实现
  • 兼顾跨本体迁移与单本体精度

核心矛盾：任务语义通用性 vs 本体执行精度 — 下面三篇论文分别从不同角度给出方案

论文支撑 1 — 深入分析

为什么 Delta 更易学习？

原始问题

图像 → 全局构型

Delta 转化

图像+状态 → 局部变化

局部变化特点：均值接近 0、方差小、分布集中、易于共享动作模式

Joint 更强的三大原因

1

避免 IK 误差

IK多解歧义即是误差来源

2

数据在关节流形上

示范轨迹包含固定肘部偏好、连续IK分支、避限位习惯

3

强生成模型可建模多模态分布

Flow Matching/Diffusion能表达多解，弱回归会平均化

⚠ 局限性
  • →EEF性能高度依赖IK和笛卡尔控制器质量
  • →跨本体实验规模小，未覆盖force/tactile/灵巧手
  • →未来可能不是二选一，而是根据任务阶段动态混合

结论

固定本体：Joint + chunk-wise delta 最优

跨本体迁移：EEF + chunk-wise delta 更适合

→ 未来方向：Reaching阶段用EEF delta，接触阶段切换Joint

三篇论文，同一套动作语言

它们为什么都不约而同选择了「以当前 EE 为锚点的相对增量」？
因为这套表示同时解决了三个核心工程问题。

UMI
2024 · Columbia / Toyota
relative EE trajectory

所有未来 pose 都相对于当前同一个 EE pose，而非逐帧递推。整段 trajectory 共享一个 anchor，模型只需学一个紧凑的局部动作分布，而非全局坐标下的复杂轨迹。

解决了什么问题：单锚点避免了逐帧递推的误差累积；动作空间均值接近零、方差小，显著降低学习难度。

GEAR-VLA
2026 · 跨本体迁移框架
Relative End-effector Action

明确强调不是相对前一预测帧，而是始终锚定当前观测帧的 EE pose。输出端保持本体无关，输入端保留关节状态作为上下文，兼顾迁移性与精度。

解决了什么问题：锚定观测帧而非预测帧，切断自回归误差链；EE 层面本体无关，同一策略可直接迁移到不同机器人构型。

GR00T N1.7
NVIDIA · 人形机器人基座模型
Relative EEF Action Space

未来动作定义为相对于 current pose 的 delta，贯穿整个预测 horizon。一个 anchor 多步输出，天然适配 chunk prediction 范式，不需要额外的动作归一化。

解决了什么问题：相对表示消除了绝对坐标系下的分布偏移；chunk 输出减少了推理次数，提升了实时控制的吞吐效率。

三种叫法，同一本质：以当前帧 EE 为锚点，预测相对增量。

IK 的角色

为什么需要 IK？

从 EEF 到算出机器人可执行的 joints 需要 IK，而 FK 就是正向过程，已知 Joints 到算出末端姿态。

而假如一个模型输出为 EEF，则需要再次从 EEF 经过特定 IK 或者控制器到真正 Joints 的映射，所以最终的实际效果不仅受限于模型的 EEF 输出，同时也取决于你用的 IK 算法的精度和有效性。

IK 的角色

现在的模型怎么做？—— 以 GR00T N1.7 DROID 为例

核心发现

以 GR00T N1.7 的 DROID 配置为例，状态和动作中同时包含：

  • relative EEF
  • relative joint position
  • gripper

而官方 DROID 真机脚本最终取 joint_position 输出执行。

关键洞察

EEF 确实可以承担跨本体表示、辅助监督和几何理解，而 Joint head 承担目标本体的稳定执行。

也就是说输入可以容纳许多字段，但是真正推理执行的时候大家都通常保留 joints 直接输出机器人关节角度。从而避免从 EEF 经过 IK 算法到 joint 的在线 IK 过程。

本质解释

它不是物理上"没有 IK"，而是：模型从示范数据中隐式学习了本体的关节运动规律，在线时不再调用一个单独的 IK 求解器。

优势

稳定、简单，没有在线 IK 分支跳变

代价

强本体绑定

1EEF 提供通用任务几何
2Joint 保留目标本体的肩肘构型
3执行时绕过显式在线 IK
4可通过 FK consistency 约束两输出一致

IK 的角色

但 IK 算法仍然是必需的

场景一：脚本采集

在脚本采集的时候通常都是设定对应的 EEF waypoint，然后通过 IK 算出对应的 joints，而不是直接写死关节角度。

场景二：数据扩充（如 MimicGen）

在类 mimicGen 的数据扩充功能中，算法本身就是基于物体的位姿 EEF 相对变化，此时更需要一个好的 IK 算法来对 joint 进行约束，避免奇异值等多种问题，从而得到更加平滑可用的轨迹。

（这里不细说具体每一个 IK 算法的特征，毕竟经典的有 PINK IK、Lula IK 等）

GEAR-VLA

arxiv.org/html/2606.08530v1 · 2026-06-07

核心设计

  • →将 q + EEF 作为输入，EEF 作为输出
  • →输出保持本体无关 → 可跨平台迁移
  • →融合了 Joint 精度与 EEF 通用性

输入保留关节状态 → 模型知道当前本体构型

与观点3的呼应

  • →EE 提供共享任务几何（跨机器人通用语言）
  • →关节状态作为上下文让模型感知本体约束
  • →解决了纯 EEF 方法无法感知构型的缺陷

实验亮点

  • →在多个真实机器人上验证跨本体迁移
  • →zero-shot transfer 性能显著优于纯 Joint 方法
  • →固定本体场景也不输给专用 Joint 策略

新训练范式提出

论文支撑 3 + 新范式

IKSel + 新训练范式

arxiv.org/html/2503.22234v1

新范式核心

VLA 输入 EEF → 输出 EEF → IKSel 选择最优种子 → 数值IK快速求解 Joint

IKSel 核心思路

  • →为快速数值IK迭代选择合适的种子关节值
  • →从多个候选种子中评估选择最优起点
  • →确保IK收敛快、解连续、避免奇异点
  • →结合 Joint 参考列表避免极端构型

完整Pipeline

  • →Step 1: VLA 模型从视觉输入预测 EEF 目标（本体无关）
  • →Step 2: IKSel 根据当前 Joint 列表选择最优 IK 种子
  • →Step 3: 数值 IK 从最优种子快速收敛到目标关节角
  • →Step 4: 输出 Joint Command 执行

总结

Core Takeaways

1

Joint 在固定本体下表现更强，但本质是照葫芦画瓢，不是终局之解

2

EEF 能学到任务本意，但忽略本体 IK 局限性

3

融合方案：EE 提供共享任务几何 + 关节状态保证可执行性（GEAR-VLA）

4

新范式：VLA 输入/输出 EEF + IKSel 快速求解 Joint，兼顾通用性与精度

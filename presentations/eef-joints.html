Terminology

基础定义：EEF 与 Joints

EEF 描述「手在哪」，Joints 描述「臂怎么放」—— IK 是两者之间存在多解歧义的桥梁

EEF (End-Effector Frame)

[Δx, Δy, Δz, Δroll, Δpitch, Δyaw, g] ∈ ℝ7

末端执行器在笛卡尔空间的 6-DoF 位姿 + 1 个夹爪开合值

Joints（关节空间）

q ∈ ℝn （G2 上半身 n = 14）

以智元 G2 为例：双臂各 7 关节，数据即关节角向量

FK（正运动学）

FK: q → EEF 位姿

确定性、唯一解

IK（逆运动学）

IK: EEF 位姿 → q

可能多解、需选择策略 —— 后文讨论的关键环节

IK's Role

IK 的角色 —— EEF 与 Joints 的桥接器

模型输出只是「提议」，IK / 控制器才是最后一道关卡，它决定哪些动作真正被执行

Fig 1. OpenVLA 架构：输出 7-DoF delta EEF action，需下游 IK/控制器映射到关节

  • 模型输出 EEF → 需经 IK 求解才能执行；执行效果同时受限于 IK 算法质量
  • 各模型 action 定义差异大：relative vs absolute、delta 锚点、EEF vs joint

Action Space · Part A

EEF 输出流派：输出的是相对当前位姿的 delta，能否绕过 IK 完全取决于配置

01 OpenVLA 7-DoF delta EEF github.com/openvla/openvla

锚定当前 EEF 位姿的增量 [Δx, Δy, Δz, Δroll, Δpitch, Δyaw, g]，不是绝对位姿

必需下游 IK：current_eef + delta → IK → joint；旋转用小角度近似简化为向量加法

action = vla.predict_action(**inputs, unnorm_key="bridge_orig")
02 OpenPI π₀ joint / EEF 可配置 github.com/Physical-Intelligence/openpi

DROID 场景输出 joint delta（7 关节 + 1 gripper）；action_dim = 8 (FAST-DROID) / 32 (π₀.₅)，缺失维度补 0

joint 模式绕过 IK；EEF 模式同 OpenVLA 需 IK；执行依赖环境侧控制器，框架不内建 IK

config = Pi0FASTConfig(action_dim=8, action_horizon=10)

Action Space · Part B

双 head / 统一向量流派：EEF 与 Joint 同时编码，真机执行取 joint 侧绕过 IK

01 GR00T N1.7 双 head：ΔEEF + Δq github.com/NVIDIA/Isaac-GR00T

两个 head 都锚定当前观测帧（不是上一预测帧）；action 定义由 embodiment tag 决定（LIBERO 为 absolute joint）

真机执行取 joint head 绕过 IK；EEF head 仅作训练监督

Three Viewpoints

核心矛盾：任务语义通用性 vs 本体执行精度

01
Joint 照葫芦画瓢 数据绑定特定机器人构型，泛化受限；工程表现好，但不是终局之解
02
EEF 学任务本意但忽略 IK 局限 能学到任务本意，但胳膊如何过去同样重要；IK 多解歧义 → 执行抖动或失败
03
EE + 关节状态融合 EE 提供共享任务几何，关节状态保证可执行性；代表：GEAR-VLA，兼顾迁移与精度

Paper Reference · Deep Dive

固定本体：Joint + chunk-wise delta 最优

跨本体迁移：EEF + chunk-wise delta 更适合

Joint 更强的三大原因

①

避免 IK 误差

IK 多解歧义即是误差来源，直接输出 Joint 绕开这一环节

②

数据本就在关节流形上

示范轨迹包含肘部偏好、连续 IK 分支、避限位习惯

③

强生成模型可建模多模态分布

Flow Matching / Diffusion 能表达多解，弱回归会平均化

局限性：EEF 依赖 IK 质量 / 跨本体实验规模小，未覆盖 force·tactile·灵巧手 / 未来可能按任务阶段动态混合而非二选一

Deep Dive · Delta

Chunk-wise delta 锚定同一观测帧，误差不累积 —— 这是领先 ~10% 的原因

01Step-wise逐步增量
at = qt − qt−1 → qt = qt−1 + at

参考上一步串行递推，误差逐步累积（drift） —— 像闭眼走路每步靠上一步，越走越偏

02Chunk-wise锚定增量
at+k = qt+k − qt → qt+k = qt + at+k

整段锚定同一观测点 qt，不累积误差、天然并行 —— 盯着起点地标走，每步独立参考

03为什么 Delta 优于 Absolute

Absolute：直接输出 qtarget，不同机器人绝对值分布差异巨大 → 需拟合全局关节空间，泛化差

Delta：输出 Δq = qtarget − qcurrent，分布紧凑（均值≈0、方差小）→ 与起始状态无关，泛化强

04EEF 的特殊性SE(3) 上的 delta 不是加法
Ttarget = ΔT · Tcurrent （位置可加，旋转是矩阵 / 四元数乘法）

⚠ EEF delta 需要群运算，工程实现比 joint delta 复杂 —— 这也是固定本体 Joint delta 更优的原因之一

Industry Consensus

三篇论文，同一套动作语言

它们为什么不约而同选择了这套表示？因为它同时解决了三个核心工程问题。

01
UMI 2024 · Columbia / Toyota · relative EE trajectory 所有未来 pose 都相对于当前同一个 EE pose，而非逐帧递推。整段 trajectory 共享一个 anchor，模型只需学一个紧凑的局部动作分布。
02
GEAR-VLA 2026 · 跨本体迁移框架 · Relative End-effector Action 明确强调不是相对前一预测帧，而是始终锚定当前观测帧的 EE pose。输出端保持本体无关，同一策略可直接迁移到不同机器人构型。
03
GR00T N1.7 NVIDIA · 人形机器人基座模型 · Relative EEF Action Space 未来动作定义为相对于 current pose 的 delta，贯穿整个预测 horizon。相对表示消除绝对坐标系下的分布偏移。

UMI · GEAR-VLA · GR00T N1.7 —— 名字不同，锚点相同。

IK Still Matters

但 IK 算法仍然是必需的

即使执行端用 Joint，IK 在数据侧也不可或缺 —— 脚本采集与数据扩充都靠它

场景一：脚本采集

设定 EEF waypoint → IK 反算 joints，而非写死关节角。

场景二：数据扩充（如 MimicGen）

算法基于 EEF 相对变化生成新轨迹，更需好 IK 约束 joint、避开奇异点，得到平滑可用的轨迹。

如何选择 IK？

不选排行榜第一，而是沿用：采集时的控制器 / 厂商验证过的笛卡尔接口 / benchmark 标准控制器

IK Summary

三个核心认知

没有好的 IK，再好的 EEF 预测也落不了地

一、EEF 到执行的必经桥梁

模型输出 EEF 后，必须经 IK 才能得到可执行关节指令；IK 质量决定执行效果。

二、数据侧不可或缺

脚本采集、MimicGen 扩充、仿真生成（Isaac Sim）都依赖 IK 从 EEF 反算平滑关节轨迹。

三、选型与速度

多解歧义需合理种子策略（如 IKSel）；高频回路要求快且不牺牲解的质量。

IK 既决定执行质量，也是数据流水线的关键一环 —— 选对 IK、又快又稳，是 EEF 方案可用的最后一块拼图。

核心结论

01

Joint 在固定本体下表现更强，但本质是照葫芦画瓢，不是终局之解

02

EEF 能学到任务本意，但忽略本体 IK 局限性

03

融合方案：EE 提供共享任务几何 + 关节状态保证可执行性（GEAR-VLA）

04

新范式：VLA 输入/输出 EEF + IKSel 快速求解 Joint，兼顾通用性与精度
